# 二刀流アプリ開発（Vibe Coding教材）企画説明書

> エンジニアの方へ：これは「やりたいこと」を非エンジニアの言葉から技術的に翻訳した説明書です。題材アプリはまだ確定していません。骨格（アーキテクチャ）の合意を取ることがこの文書のゴールです。

---

## 1. 一言でいうと

**「ボタンで自分で操作する道」と「AIと会話して同じことをやってもらう道」が両方あって、両者が同じ処理を呼ぶ ―― そういう"二刀流"のWebアプリを、ハンズオン教材として作りたい。**

前回（Twitter Clone）の次のステップとして、AIを「会話の相手」ではなく「アプリを操作するエージェント」として組み込む設計を学ぶことが目的です。

---

## 2. なぜこれをやりたいか（背景）

NotebookLMやSupabaseを触ったときの体験がきっかけです。

- Supabase では、RLSポリシーを **SQL（コード）で書いてもいい** し、**GUIのポリシー設定画面でポチポチ作ってもいい**。どちらも最終的に同じ結果に到達する。
- NotebookLM では、AIと会話しながら、画面上の既存のボタン（スタジオ等）が連動して動く。

この「**同じ機能に、直感的なGUIという入口と、AI会話という入口の両方からアクセスできる**」という構造そのものを、自分の手で作って理解したい、というのが要望の核です。

直感的に操作したい人はボタンを押せばいいし、ざっくり指示したい人はAIに言えばいい。利用者がどちらの入口を選んでも、裏側は同じ処理が走る ―― これを実装レベルで体験するのが教材の狙いです。

---

## 3. やりたいことの本質（技術的な翻訳）

非エンジニアの言葉を、設計の言葉に翻訳すると次のようになります。

| 元の言葉 | 技術的な意味 |
|---|---|
| 「ボタンで直感的に操作する道」 | GUIイベント（onClick等）が**アクション関数**を呼ぶ |
| 「AIと会話して操作してもらう道」 | LLMの **Tool Use / Function Calling** が、同じ**アクション関数**を呼ぶ |
| 「どちらでも同じことが起きる」 | GUIとAIが**共通のアクション層**を経由する |
| 「会話から得られるコーディング」 | 自然言語 → 構造化された関数呼び出し への変換をLLMに任せる |

つまり設計の肝は **「操作ロジックを共通のアクション層に集約し、GUIとAIの両方がそれを叩く」** という一点に集約されます。

---

## 4. アーキテクチャ（最重要）

```
  ┌──────────────────┐        ┌──────────────────┐
  │  道①: GUI操作     │        │  道②: AIチャット   │
  │  ボタン / フォーム  │        │  自然言語入力      │
  └────────┬─────────┘        └─────────┬────────┘
           │                            │
           │                  ┌─────────▼─────────┐
           │                  │  LLM (Tool Use)    │
           │                  │  言葉→関数呼び出しに │
           │                  │  翻訳               │
           │                  └─────────┬─────────┘
           │                            │
           ▼                            ▼
  ┌────────────────────────────────────────────┐
  │           共通アクション層                     │
  │   addItem() / updateItem() / deleteItem() …  │
  │   （GUIもAIも、必ずここを通る）                 │
  └────────────────────┬───────────────────────┘
                       │
                       ▼
              ┌─────────────────┐
              │   データベース     │
              └─────────────────┘
```

**設計原則：**

1. **操作はすべて「アクション関数」として定義する。** UIコンポーネントやAIハンドラの中に直接DB操作を書かない。
2. **GUIのイベントハンドラは、アクション関数を呼ぶだけ。**
3. **AIに渡すTool定義（スキーマ）は、アクション関数と1対1で対応させる。** ツール名・引数を関数シグネチャに揃える。
4. **LLMの役割は「翻訳」だけ。** ユーザーの自然言語を「どのツールを、どの引数で呼ぶか」に変換する。実際のDB操作はアクション層が担う。

この原則を守ると、機能を1つ追加するたびに「アクション関数を1つ書く → GUIボタンを足す → Tool定義を足す」という綺麗な対応関係で拡張できます。教材として「同じパターンの繰り返しで身につく」状態になります。

---

## 5. AIチャット側の処理フロー

道②（AIチャット）が1回の指示を処理する流れ：

1. ユーザーが自然言語で入力（例:「未着手のタスクを全部完了にして」）
2. アプリが、ユーザー入力 ＋ 利用可能なTool定義一覧 をLLMに渡す
3. LLMが「どのツールを、どの引数で呼ぶべきか」を返す（`tool_use`）
4. アプリが、その指示どおりに**共通アクション層の関数を実行**する
5. 実行結果をLLMに返し、ユーザー向けの自然な返答を生成させる
6. 画面（GUI側）も同じデータを見ているので、自動的に表示が更新される

> ポイント：LLMは直接DBを触らない。LLMは「どの関数をどう呼ぶか」を決めるだけで、実行権限はアプリ側が握る。これが安全面でも実装面でも重要。

## 6. 想定技術スタック

前回（Twitter Clone）の資産を最大限に再利用する前提です。

| レイヤー | 技術 | 前回からの差分 |
|---|---|---|
| フレームワーク | Next.js (App Router) + TypeScript | 再利用 |
| スタイル | Tailwind CSS | 再利用 |
| 認証 | Google OAuth | 再利用 |
| DB / バックエンド | Supabase | 再利用 |
| **AI操作** | **Claude API（Tool Use / Function Calling）** | **★今回の新規学習ポイント** |
| GUI操作 | Reactの標準イベント（onClick等）、必要に応じてドラッグ系ライブラリ | 一部新規 |

学習負荷の見積り：**約8割は前回の再利用、新規学習は「共通アクション層の設計」と「Tool Use」の2点に集中**。

---

## 7. この教材で学べること（学習ゴール）

- [ ] 操作ロジックをUIから分離し、共通アクション層に集約する設計
- [ ] LLMのTool Use（Function Calling）の仕組みと実装
- [ ] 自然言語の指示を、安全に構造化された関数呼び出しへ変換する流れ
- [ ] GUIとAIという2つの入口を、1つの状態（データ）に対して両立させる設計
- [ ] AIを「会話相手」ではなく「アプリのオペレーター」として組み込む考え方

---

## 8. まだ決まっていないこと（要相談）

- **題材アプリの確定**：上記アーキテクチャは題材に依存しない。「データを追加・更新・削除する」タイプのアプリなら何でも当てはまる。題材は、学習者がイメージしやすく、かつ操作が二刀流で自然に成立するものを選びたい。
- **AI操作の範囲**：データ操作（追加・編集・削除）に絞るか、検索・絞り込み・並べ替えも含めるか。
- **GUI操作の作り込み度**：シンプルなボタン中心にするか、ドラッグ操作など凝った操作も入れるか。

---

## 9. エンジニアへの依頼事項

この説明書を読んだうえで、次の点について意見がほしいです。

1. このアーキテクチャ（共通アクション層をGUIとAIが共有する構成）は妥当か。改善点はあるか。
2. Tool Use の実装で、教材としてつまずきやすいポイントはどこか。
3. 題材アプリとして、この骨格を活かしやすい候補があれば提案してほしい。
